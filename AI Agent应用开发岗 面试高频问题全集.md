## 二、实习经历（搜狐Agent研发实习生）深挖类

### （一）数据采集与反爬相关

###### 1. 你在搜狐实习中累计爬取31万+条结构化资讯，具体是爬取哪些平台的信息？爬取的频率和调度策略是怎样的？

我这个项目分三期爬取，一期懂车帝 84 个汽车广告主，二期扩展易车、汽车之家，三期扩展头条、腾讯新闻共 1544 个全行业广告主，累计爬取 31 万 + 结构化数据，字段包括标题、简述、发布时间、链接、广告主、平台和爬取时间。调度上采用初始化全量爬取，后期按平台差异化定时增量更新，配合分片、限速、重试和去重策略，保证稳定运行。

###### 2. 你实现了重试策略、Cookie池切换、滑块验证绕过等反爬机制，具体说说每种反爬机制的技术实现细节？比如滑块验证是通过什么方式绕过的（图像识别/模拟轨迹/第三方工具）？

我们采用三层防御：指数退避重试（3s→6s→12s加随机抖动）、多账号Cookie池自动轮换、以及Cookie持久化绕过滑块验证——首次人工完成验证后保存Cookie，后续自动加载复用，配合undetected-chromedriver和多节点Selenium Grid实现稳定高效的反爬对抗。

###### 3. 数据采集成功率做到95%以上，剩下5%的失败原因主要是什么？有没有尝试过进一步优化？

剩余5%主要由三类原因导致：目标站点页面结构变更导致解析失败（约2%）、极端风控触发账号封禁（约2%）、网络超时等偶发异常（约1%）；优化方面我们做了动态选择器配置和异常自动重试，但页面结构变更需要人工介入更新规则，这是平衡维护成本后的取舍。

###### 4. 断点续爬的功能是如何实现的？是基于什么标识记录爬取进度的？如果遇到网站结构变更，断点续爬会如何适配？

基于关键词级别的断点续爬：以**关键词**为粒度记录进度，每个关键词完成爬取后实时写入 `{site}_progress.txt` 文件，记录已完成和待处理关键词列表；重启时读取进度文件，跳过已完成的关键词继续执行。若遇网站结构变更，需人工更新解析规则后重新爬取受影响的关键词——我们采用**关键词粒度**而非页面粒度，正是为了在结构变更时能以最小成本重置和重试。

###### 5. 爬取的1677个广告主数据是如何分类和筛选的？如何保证广告主数据的时效性和准确性？

广告主数据主要来自**关键词文件**（如 `keywords_toutiao.txt` 包含约1600个品牌/企业名称），分类上按**业务平台维度**划分（今日头条、腾讯新闻、懂车帝、汽车之家、易车网），筛选通过**广告主名称去重**实现（`remove_duplicates.py` 按广告主字段保留首次出现记录）；时效性方面，数据通过**定期爬取**更新，准确性依赖**多源交叉验证**（同一广告主在多个平台爬取对比）和**人工抽检**，但项目代码中未显示自动化的时效性监控机制，主要靠任务调度定期执行来保证数据新鲜度。

###### 6. 爬取过程中有没有遇到过IP封禁、法律合规相关的问题？是如何规避的？

IP封禁方面，项目通过**多账号Cookie轮换**、**指数退避重试**和**Selenium Grid多节点**分散请求压力来规避，但代码中未使用代理IP；法律合规方面，项目主要采集**公开搜索结果的标题和摘要**（非全文），并通过**限速控制**（关键词间延迟5秒、批次间延迟10秒）降低对目标站点的压力。

###### 7. 你在爬虫模块开发中，用的是Python的哪些库？除了Selenium、BeautifulSoup4，有没有考虑过Scrapy、Playwright等框架？为什么选择当前的技术选型？

核心库用了 **Selenium + undetected-chromedriver** 做浏览器自动化，**requests** 做简单HTTP请求，解析主要用正则和字符串处理，没有使用 BeautifulSoup4；框架选型上**考虑过 Scrapy 和 Playwright**，最终选择当前方案是因为：Scrapy 的异步架构虽然效率高，但我们的场景需要处理大量 JavaScript 渲染和滑块验证，Selenium 更直接可控。

### （二）数据处理与知识库建设相关

###### 1. 你构建的数据清洗与向量化管道，具体的清洗步骤是什么？如何处理脏数据、重复数据、无效数据？

把爬下来的数据里的**JS 乱码过滤掉**，**重复的内容删掉**，**缺标题、缺链接或者链接打不开的无效数据剔除**，再把时间、链接格式统一整理干净，最后把这些高质量数据送入 FastGPT 做向量化，构建知识库。

###### 2. 爬取的资讯数据是如何做向量化的？选择了哪个嵌入模型（Embedding Model）？为什么选择这个模型？向量化的维度和参数是如何调优的？

数据向量化是把清洗后的标题和资讯简述拼接后，通过 **bge-large-zh-v1.5** 模型生成向量。选择它是因为该模型在中文语义检索上效果稳定、开源易用，行业应用成熟。向量维度采用模型默认的 **1024 维**，实际使用中主要通过调整检索的 top-k 和相似度阈值来优化效果，保证召回精准。

###### 3. 处理后的数据存入MySQL，具体的表结构是如何设计的？针对大规模数据（31万+），做了哪些SQL优化来提升查询效率？

表结构就是一张舆情资讯表，存储标题、简述、发布时间、链接、广告主、平台、爬取时间这些字段。针对 31 万 + 数据，我主要做了三个优化：第一，给广告主、平台、发布时间建立**联合索引**；第二，查询时只返回需要的字段，不使用 SELECT；第三，限制时间范围查询，避免全表扫描。最终接口响应稳定在 200ms 以内。

###### 4. 数据向量化后存入知识库，这个知识库是基于什么框架/工具搭建的？和MySQL的数据是如何做同步的？

我们的知识库是基于 **FastGPT** 搭建的，底层使用向量数据库存储向量化数据。同步方面，初期通过 CSV 批量导入；后期接入 MySQL 后，使用**定时任务增量同步**，定期把新数据自动写入 FastGPT，保证知识库和数据库始终保持一致。

###### 5. 你说竞品追踪效率提升70%，报告周期从3天缩至4小时，具体是怎么量化这个效率提升的？Agent是如何自主完成检索与归纳的？检索的精准度如何保证？

之前人工做竞品追踪，一份报告要 3 天，现在用系统 4 小时就能完成，报告周期从 3 天缩短到 4 小时，效率提升约 70%，这是通过人工工作量和自动化处理量对比量化的。Agent 可以自主理解销售需求，自动调用接口检索数据，筛选后自动归纳生成报告，全程无需人工干预。检索精准度主要靠严格的查询匹配、干净的数据源、以及向量相似度阈值控制来保证。

### （三）后台接口开发与业务支撑相关

###### 1. 为什么选择Go+Gin开发后台接口，而不是继续用Python的FastAPI等框架？Go相比Python在这个场景下的优势是什么？

技术栈是公司统一要求的。Go 相比 Python 优势很明显：**性能更好、接口更快**，而且公司内部已经有成熟的 Go 脚手架和基建，部署简单、稳定性强，更适合后端服务长期运行。

###### 2. 接口响应做到<200ms，日调用峰值5万次，你做了哪些性能优化？比如接口层面、数据库层面、缓存层面？

主要就是给 MySQL 建了联合索引避免全表扫，高频数据放 Redis 缓存，再加上 Go 本身性能好，接口字段也精简了，所以轻松压到 200ms 以内，扛住 5 万次日调用峰值。

###### 3. 接口的功能主要是哪些？如何设计接口的权限控制？支撑8个销售小组，有没有做过接口的个性化定制？

我开发的后台接口主要用于广告舆情数据查询，支持按平台、广告主、时间范围筛选并返回内容与跳转链接，一方面支撑前端页面供 8 个销售小组使用，另一方面也对外开放给内部 Agent 自动调用；权限上做了角色分离，销售和 Agent 只有查询调用权限，管理员可进行数据管理，保证接口安全。

###### 4. 接口的监控和容灾机制是怎样的？如果接口出现故障/响应超时，会如何处理？

监控和容灾都是公司 Gin Web Starter 脚手架内置的，包含接口监控、日志、告警和熔断限流；接口超时或故障会自动告警并熔断降级，保障销售端和 Agent 调用稳定。

###### 5. 这个接口和Agent知识库是如何打通的？销售团队的提问是如何传递到Agent，又如何通过接口获取数据的？

前期我是把爬取的数据生成 CSV，通过同事给的脚本上传到 FastGPT 构建知识库。后期数据存入 MySQL 后，我们通过定时任务增量同步到 FastGPT 向量库，实现知识库自动更新。销售提问会先到 FastGPT 驱动的 Agent，Agent 解析意图后，调用我开发的后台接口从 MySQL 查实时舆情数据，再整合回答返回。

## 三、项目经历拆解类

### （一）项目1：安全测试智能体（secbot）

###### 1. 请详细介绍一下你设计的Planner–Coordinator–Specialist–Summary多智能体架构，每个角色的核心职责是什么？角色之间的通信和协作机制是怎样的？

这套架构本质是分层调度，每层职责单一：Planner 做意图分类，把技术任务拆成带 `agent_hint` 的 Todo 列表；Coordinator 作为统一入口负责路由，按 `agent_hint` 分发给对应专职 Agent；五类 Specialist（NetworkRecon/WebPentest/OSINT/TerminalOps/DefenseMonitor）各带领域工具集做 ReAct 执行；SummaryAgent 汇总结构化结果生成 Markdown 报告。通信上整条链路通过 EventBus 解耦，各阶段事件广播，前端 SSE 实时订阅，Agent 之间不直接调用，由 TaskExecutor 驱动 Todo 队列逐步执行。

###### 2. 单次任务可稳定调度10+子智能体协同执行，调度的策略是什么？如何解决子智能体之间的资源竞争、任务冲突问题？

调度策略是依赖分层 + 串并行混合：`get_execution_order()` 分析依赖关系，有先后依赖的 Todo 串行，同层无依赖的 Todo 用 `asyncio.gather` 并发。资源竞争靠任务隔离规避——每个 Todo 有独立 context 字典，结果按 `todo_id` + `resource` 分维度存储，互不写冲突；某个 Todo 抛异常用 `return_exceptions=True` 捕获，标记失败但不中断整条链。坦白说，目前对系统级资源（带宽、并发进程数）还没有信号量保护，这是待优化点。

###### 3. 引入的TaskExecutor分层并发执行与工具路由机制，具体的实现细节是什么？分层是按什么维度分的？工具路由是如何匹配子智能体和对应的工具的？

分层由三个条件共同决定：`depends_on` 有依赖的 Todo 必须等前置完成、同一 IP 且 `risk_level=high` 的 Todo 强制串行、每层最多 3 个 Todo 并行（超出顺延下层）。工具路由是三级匹配：优先命中 Planner 预填的 `agent_hint`，其次按 `resource` 前缀兜底（`host:` → NetworkRecon，`web:` → WebPentest），再次按 `tool_hint` 关键词兜底，三级都没命中回退默认 HackbotAgent；`resource` 字段本身也是自动从 Todo 内容里用正则推断的。

###### 4. 集成了10+ LLM供应商与HTTP回退策略，为什么要集成多个LLM供应商？HTTP回退策略是如何设计的？比如某个LLM调用失败，会如何切换？

多供应商是把选型权交给用户：国内用户用 DeepSeek/通义/智谱，想免费的用 Groq/OpenRouter，隐私敏感的用 Ollama 本地。HTTP 回退是两层降级：正常走 LangChain `ainvoke`，捕获到 `model_dump` / `generation chunks` 等兼容性异常后，`llm_http_fallback.py` 直接用 `httpx` 向 `/v1/chat/completions` 发 POST 绕过 LangChain 层——注意这不是跨供应商切换，仍是同一个供应商，只是把调用层从 SDK 降级为原始 HTTP。连 HTTP 直连也失败才返回带前缀的错误字符串。

###### 5. 核心智能体调用成功率提升至95%+，提升成功率的核心优化点有哪些？除了回退策略，还有哪些措施？

几个关键点：LLM 调用有三级降级链（流式→非流式→HTTP直连），每级独立 catch；`asyncio.wait_for(timeout=30.0)` 包裹所有调用防止挂死；`_parse_action()` 用括号深度匹配而非简单正则解析 Action JSON，处理嵌套花括号等不规范输出；工具找不到时做子串模糊匹配减少大小写差异失败；参数提取失败直接用空 `{}` 继续而不是中断；每个 Agent 有并发锁，同一时间只处理一个任务；工具执行失败结果作为 Observation 写回历史，LLM 下轮可以换工具重试。

###### 6. 基于TypeScript + Ink构建终端TUI的原因是什么？TUI的核心功能有哪些？相比传统的Web界面，终端TUI在安全测试场景下的优势是什么？

选 Ink 是因为它把 React 组件模型搬到终端，Flexbox 布局 + hooks，和已有的 React Native 代码同一套思路，学习成本低；TypeScript 在对接复杂 SSE 数据流时编译期就能发现问题。核心功能：SSE 流式渲染 Agent 思考过程、斜杠命令面板（`/task`/`/model`/`/tools` 等）支持模糊搜索、Dialog Stack 弹窗系统、ask/task 双模式切换、自适应终端尺寸。安全测试场景优势：SSH 远程环境开箱即用无需浏览器、不占额外端口不留进程痕迹、输出可直接 pipe 给 nmap/grep、不需要监听 0.0.0.0 减少暴露面。

###### 7. 通过FastAPI/SSE打通后端编排与前端会话视图，SSE在这个场景下的作用是什么？相比WebSocket，为什么选择SSE？

Agent 执行链路长（规划→多步 ReAct→工具→汇总），SSE 把整个过程事件化实时推送：后端 EventBus 订阅所有内部事件（`PLAN_START`/`THINK_CHUNK`/`EXEC_RESULT`...），映射成 SSE 帧推出，前端 `ReadableStream` 消费按类型渲染。还有个特殊用法：涉及 sudo 时后端暂停并推送 `root_required` 事件，用户 TUI 弹窗确认后 POST 回来解除阻塞，单条 SSE 连接里完成了一次双向交互。选 SSE 不选 WebSocket：通信方向天然单向、FastAPI 几十行搞定、走标准 HTTP/1.1 对代理防火墙更友好，WebSocket 在这里是过度设计。

###### 8. 人工干预减少约40%，具体是哪些环节减少了人工干预？如何量化人工干预的减少？

主要三个环节：信息收集阶段 nmap/whois 等工具链由 Agent 自动拆分执行、报告生成由 SummaryAgent 自动汇总、工具参数由 Agent 从上下文自动提取，不再需要手动记各种 flag。量化方式是统计完成同类任务从"手动步骤数"到"一句话触发后等结果"的操作次数对比。

###### 9. 新用户10分钟内可完成配置并跑通首个渗透任务，你在产品设计和文档层面做了哪些优化来降低使用门槛？

默认走 Ollama 本地模型，四步启动（`uv sync` → `ollama pull` → 复制 `.env` → `python main.py`）无需注册账号；模型切换和 API Key 配置内聚到 `/model` 弹窗，改完即生效存 SQLite，不需要手动改文件；QUICKSTART 严格线性步骤，配了实际运行截图；TUI 内置 `/help`/`/tools`/`/list-agents`，输入 `/` 就能探索全部能力，不需要读完整文档。局限是文档项目结构部分未及时更新，低配机器推理慢体验会打折扣。

###### 10. 这个项目的应用场景是什么？目前有实际的用户/团队在使用吗？收到的反馈是什么？还有哪些待优化的点？

定位是授权范围内的自动化渗透测试辅助，主要用于靶机练习、内网信息收集自动化、安全学习者自然语言驱动工具。目前 GitHub 开源，有 star 和 fork，但还是早期阶段，主要是个人开发者和安全爱好者试用。正向反馈集中在流式推理可见、多 LLM 支持、Ollama 零费用；问题反馈集中在小模型工具调用格式不规范导致解析失败。待优化：小模型 function calling 标准化适配、并行时系统级资源限制、可视化任务进度、历史任务回放。

###### 11. 项目基于LangChain/LangGraph开发，LangGraph相比纯LangChain，在多智能体架构设计中的核心优势是什么？

先说实情：项目对 LangGraph 是可选依赖，没装自动降级为纯 Python 状态机。LangChain 本质是线性管道，没有原生条件路由和循环；LangGraph 引入 StateGraph，优势在于：条件路由（验证成功走下一步/失败走回滚）、图有环路天然支持迭代推理、所有节点共享 state 字典。在这个项目里实际用于攻击链（初始化→选步骤→执行→验证→继续/回滚），这种需要循环+条件跳转的场景用 LangGraph 比手写状态机更直观。但多 Agent 调度（TaskExecutor + CoordinatorAgent）是自己实现的，没依赖 LangGraph，它更多用于单条复杂推理链而非多 Agent 编排层。

###### 12. 安全测试智能体在执行渗透任务时，如何保证操作的合规性和安全性？会不会出现误操作的情况？如何规避？

合规性上：AuthorizationManager 支持对目标 IP 添加授权记录（带类型和过期时间），执行前检查，但坦白说目前没有强制串联到每次工具调用，更多是辅助记录；每个工具有 `sensitivity` 属性，`SuperHackbotAgent` 模式下 `high` 级工具必须用户 `/accept` 确认；涉及 sudo 的命令默认策略是 ask，通过 SSE 弹窗显式授权，不允许静默提权；每次 thought/action/observation 写入审计日志可完整回溯。误操作规避：高危操作强制确认是代码层硬约束、工具失败结果作为 Observation 反馈让 LLM 自主重试、ReAct 循环有 `max_iterations` 上限防无限执行。

### （二）项目2：多智能体RAG系统（advanced-rag）

###### 1. 请对比一下这个项目的Planner/Coordinator/Expert/Summary式Agent workflow和安全测试智能体的架构，核心区别是什么？为什么要做这样的设计差异？

RAG 的 Agent 按能力类型拆分（检索/解释/代码/公式），可以并行执行再汇总，因为各专家输出是互补的知识片段；而安全测试智能体按测试阶段拆分（侦察/扫描/验证），必须流水线顺序执行，因为后一步依赖前一步的结果——本质上是"问题本身的并行性"决定了架构差异。

###### 2. 工作流编排器顺序调度8+专家代理并流式汇总，流式汇总的实现方式是什么？如何保证汇总结果的逻辑性和完整性？

用 AsyncGenerator 逐块 yield 结果，通过 SSE 推送到前端，同时 Coordinator 先规划只选必要的 Agent、每个 Expert 拿到明确的子任务描述、单个失败不影响其他、最后 ResponseBuilder 统一格式组装，这样既保证分工明确又能让用户实时看到进度。

###### 3. 你实现的混合分块（规则保留代码/公式/表格 + 语义分块），具体的规则分块是如何设计的？语义分块用的是什么算法/模型？分块的大小和重叠度是如何调优的？

规则分块用正则提取代码块、公式、表格保持完整，剩余普通文本用 LangChain SemanticChunker + Ollama 嵌入按语义相似度断开；参数上 chunk_size=1000、overlap=200 是在"上下文完整性"和"检索精度"之间折中，语义阈值 0.5 则控制断点敏感度。

###### 4. 双路索引是Qdrant向量库 + Neo4j图谱，为什么选择这两个工具做索引？向量库和图谱分别存储什么类型的知识？二者是如何协同工作的？

选 Qdrant 是因为开源高性能、支持 gRPC 连接复用，选 Neo4j 是因为图查询成熟、适合实体关系建模；向量库存文本块嵌入做语义检索，图谱存实体-关系三元组做关联推理，入库时同时写两边，检索时并行查再融合，互补覆盖"找相似"和"找关系"两类需求。


###### 5. 向量/关键词/图谱三路并行检索，具体的检索流程是什么？如何对三路检索的结果进行融合？top_k=100、score_threshold=0.7的参数是如何确定的？

三个通道会并行做向量检索（Qdrant）、关键词检索（MongoDB 限定文档）和图谱检索（Neo4j 一跳邻居），然后按 chunk_id 把向量结果当底座、命中关键词的结果在原分数上做加权提升、图谱结果作为额外知识块直接加入，最后可选用 BGE-reranker 重排并动态裁剪返回数量，而 top_k≈100 和 score_threshold≈0.7 是在带标注的数据集上用评估脚本反复试参后，在召回率、相关性和延迟之间折中的经验值。

###### 6. BGE-reranker重排的作用是什么？开启重排后，检索精准度和效率有哪些变化？有没有做过重排与否的对比测试？

BGE-reranker就是在三路检索出一堆候选之后，用一个更“挑剔”的交叉编码模型重新给“问题-文段”配对打分并按相关性重排，实测打开后前几条命中更准、噪声更少但延迟会多几十到几百毫秒，所以项目里专门写了 `retrieval_eval.py` 做过“开/关重排”的 recall 和 precision 对比，确认是“多一点时间换明显精度提升”的结果。

###### 7. 多知识空间并行检索与检索失败回退无上下文，检索失败回退的策略是什么？无上下文回退的优势是什么？

检索失败时直接跳过检索环节，让 LLM 纯靠自己的知识回答，好处是用户体验不断、不会因为知识库挂了就整个对话崩掉，同时也方便对比"有检索 vs 无检索"的效果差异。

###### 8. 核心检索链路通过/health/metrics监控延迟与错误率，具体监控了哪些指标？如何基于这些指标做告警和优化？

主要监控检索耗时、响应时间、首 token 延迟、召回条数、上下文长度，前端会根据阈值（如检索>阈值、响应>阈值、召回<3条）自动生成告警提示，后端通过 `/health/metrics` 暴露请求统计和系统资源，运维可以据此做熔断、扩容或参数调优。

###### 9. 如何降低Agent的幻觉和超时风险？除了检索层面的优化，还有哪些措施（如Prompt工程、LLM选型、任务切分等）？

幻觉方面用检索结果约束回答、Prompt 里强调"只基于上下文回答"、来源追溯让用户能核查；超时方面用异步流式输出、LLM 调用加超时兜底、长任务拆成多步 Agent 并行执行，同时选更快的模型做简单任务、推理模型只用于复杂分析。

###### 10. 基于Next.js + TypeScript与FastAPI/SSE实现端到端链路，具体的前后端交互流程是什么？多格式解析支持哪些格式（如PDF/Word/Excel/Markdown）？用的是什么解析库？

前端通过 SSE 长连接订阅后端事件流，后端用 FastAPI 的 `EventSourceResponse` 把 Agent 状态、检索结果、生成内容逐帧推送，前端实时渲染；解析支持 PDF、Word、Markdown、TXT、图片，可选 Unstructured 做 PPT/Excel/HTML，PDF 用 PyPDF2+OCR、Word 用 python-docx、Markdown 用 markdown 库。

###### 11. 知识抽取的环节具体做了什么？抽取了哪些类型 of 知识（如实体/关系/属性）？用的是什么方法（大模型抽取/规则抽取）？

用 LLM 从文本里抽"实体-关系-实体"三元组，实体类型包括 Concept/Technology/Person/Organization 等，关系由模型自动生成，抽完直接写入 Neo4j 图谱，用于后续的关联推理检索。

###### 12. 你编写的评测脚本，将检索–生成–LLM-as-a-Judge串联，具体的评测流程是什么？0–1分制的评分维度有哪些？如何保证评测结果的客观性？

流程是：对数据集每条问题先检索上下文、再让 LLM 生成回答、最后用另一个 LLM 当评委对比标准答案打 0-1 分；评分维度主要是正确性和完整性，保证客观性的方式是用统一 prompt、固定模型、分数归一化处理，同时保存每条结果方便人工复核。

###### 13. 通过评测脚本优化Agent策略与Prompt方案，举1个你通过评测发现问题并优化的案例？优化后效果提升了多少？

比如早期检索 top_k 太小导致召回不足、回答经常漏关键信息，评测分数只有 0.6 左右，后来把 prefetch_k 从 20 调到 50、加上重排，平均分涨到 0.78，这就是"数据驱动调参"的典型例子。

###### 14. 这个多智能体RAG系统和传统的单智能体RAG系统相比，核心优势是什么？在哪些场景下提升效果最明显？

核心优势是"专业分工+并行协作"，每个 Agent 只做一件事（检索、解释代码、分析公式），Coordinator 按问题类型动态选人，比单 Agent 一把梭更精准；在复杂问题（如"解释这段代码的物理公式"）需要多维度知识时提升最明显。

###### 15. 项目中用到了Qdrant向量库，你对Qdrant的底层原理有哪些了解？做了哪些调优来提升Qdrant的检索效率？

Qdrant 底层用 HNSW 图索引做近似最近邻搜索，空间复杂度和查询速度都比暴力搜索优；调优方面主要是用 gRPC 连接复用降低开销、根据实际向量维度自动创建/重建集合、设置合理的 score_threshold 过滤噪声、用 filter_conditions 按文档 ID 缩小搜索范围。

## 四、技术栈专项考察类

### （一）LangChain/LangGraph相关

###### 1.请说说你对LangChain核心组件的理解（如PromptTemplate、Chain、Agent、Tool、Memory等）？在你的项目中是如何运用这些组件的？

PromptTemplate 做提示词模板化管理（变量注入、系统/用户角色分离）；Chain 把多个处理步骤串联成管道；Agent 是在 LLM 推理 + 工具调用上包了一层 ReAct 循环；Tool 是 Agent 能调用的外部能力单元；Memory 存储对话历史让模型有上下文感知。项目里主要用 PromptTemplate 管理各 Agent 的系统提示、用 Tool 封装端口扫描/漏洞检测等安全工具、用 `ChatMessageHistory` 做短期对话记忆，LangGraph 用于攻击链的状态图管理。

###### 2. LangGraph的核心概念（如State、Node、Edge、Graph）是什么？在多智能体架构中，如何通过LangGraph实现状态管理和任务流转？

State 是贯穿整条图的共享字典，所有节点读写同一份状态；Node 是一个处理函数，接收 State 返回更新后的 State；Edge 定义节点间的流转路径，分普通边（固定跳转）和条件边（运行时根据 State 值决定走哪条）；Graph 把这些组装成有向图并编译执行。在项目攻击链里，`RECON_NODE` → `SCAN_NODE` → `EXPLOIT_NODE` → `VERIFY_NODE` 是普通边，`VERIFY_NODE` 根据验证结果走条件边决定继续还是回滚，State 里存着目标信息、已执行步骤、漏洞发现等，整条链共享不需要参数透传。

###### 3. 你熟悉ReAct、Plan-and-Execute等设计范式，说说这两种范式的核心区别？分别适用于哪些AI Agent场景？你在项目中是如何选择的？

ReAct 是"边想边做"——每一步 Thought → Action → Observation，适合任务边界不清晰、需要根据工具结果动态调整的场景，比如对话式问答、单步工具调用。Plan-and-Execute 是"先规划再执行"——先生成完整 Todo 列表，再按计划逐步执行，适合任务边界明确、步骤较多、需要并发调度的场景，比如渗透测试流程。项目里两者都用：Specialist Agent 内部用 ReAct 做单步推理，整体任务流用 Plan-and-Execute 让 Planner 先拆解、TaskExecutor 再调度，这样兼顾了单步灵活性和整体可控性。

###### 4. LangChain中Agent的工具调用机制是怎样的？你在项目中是如何自定义工具的？如何处理工具调用失败的情况？

LangChain Agent 在每轮 ReAct 中，LLM 输出包含 `Action: tool_name` 和 `Action Input: {...}` 的结构化文本，框架解析后找到对应 Tool 对象执行，把结果作为 `Observation` 拼回历史，驱动下一轮推理。项目自定义工具继承 `BaseTool`，实现 `name`/`description`/`_run`/`_arun` 四个属性，`description` 写清楚"什么时候用、参数是什么"，是 LLM 能不能正确选工具的关键。工具调用失败处理：捕获异常后把错误信息格式化成 Observation 写回历史（而不是直接抛出），LLM 在下轮可以选择重试或换工具；同时工具有 `sensitivity` 属性，高危工具需要用户确认，减少因参数错误导致的误操作。

###### 5. LangChain的Memory模块有哪些类型？你在项目中用到了哪种？为什么选择这种？如何处理长对话的记忆问题？

主要类型：`ConversationBufferMemory`（完整保留所有对话）、`ConversationBufferWindowMemory`（只保留最近 k 轮）、`ConversationSummaryMemory`（用 LLM 摘要压缩历史）、`VectorStoreRetrieverMemory`（向量检索相关历史）。项目里短期用 `ChatMessageHistory` 保留当前会话的 Thought/Action/Observation，长期用自建的 EpisodicMemory（写入 SQLite）和 LongTermMemory（向量检索）。长对话处理：超过一定轮数后截断最早的历史只保留近期 + 系统提示里的任务摘要，避免 context 超出模型窗口；跨会话的重要信息则写入长期记忆，下次对话通过向量检索召回。

###### 6. 你觉得LangChain/LangGraph目前存在哪些痛点？在你的项目中是如何规避这些痛点的？

主要三个痛点：一是 LangChain 版本迭代频繁、接口变动大，项目里用 `try/except` 做兼容兜底，关键调用路径同时保留 SDK 调用和 HTTP 直连两条路；二是 LangChain 对国内 LLM 供应商的适配参差不齐，表现为 `model_dump` 错误、流式输出格式不一致，项目里实现了 `llm_http_fallback.py` 直接走原始 HTTP 绕过 SDK 层；三是 LangGraph 调试体验差，图执行过程不透明，项目里通过 EventBus 把每个节点的进入/退出事件都广播出来，配合审计日志实现完整可观测。

### （二）RAG与向量数据库相关

###### 1. 请说说RAG的核心流程是什么？从文档加载到生成回答，每个环节的核心优化点有哪些？

核心流程：文档加载 → 分块 → 向量化 → 存入向量库 → 用户提问时检索 top-k → 拼入 Prompt → LLM 生成回答。各环节优化点：分块要保留语义完整性（规则保护代码/表格 + 语义断句）；向量化选对语言匹配的嵌入模型（中文用 bge）；检索用混合策略（向量 + 关键词 + 图谱并行）；检索后加 reranker 重排提精度；Prompt 里明确"只基于上下文回答"约束幻觉；生成后可加来源追溯让用户核查。

###### 2. 你用过Qdrant向量库，还了解哪些主流的向量数据库（如Milvus/Pinecone/Chroma）？它们之间的核心区别是什么？

Qdrant 开源、支持 gRPC、过滤条件灵活，适合中小规模自部署；Milvus 支持亿级向量、分布式架构、运维成本高，适合大规模生产；Pinecone 全托管零运维、按量计费，适合快速上线不想管基础设施；Chroma 轻量纯 Python、嵌入式启动，适合原型开发和本地测试。项目选 Qdrant 是因为开源可控 + gRPC 性能好 + 支持按文档 ID 过滤，适合多知识空间隔离的场景。

###### 3. 向量数据库的核心索引算法有哪些（如HNSW/IVF_FLAT/FAISS）？你对这些算法有哪些了解？在项目中是如何选择的？

HNSW 是分层的小世界图，查询时从顶层快速定位再下沉精确搜索，速度快、精度高，但建索引慢、内存占用大；IVF_FLAT 先做聚类把向量分桶，查询时只搜最近几个桶，内存友好但召回率受 nprobe 影响；FAISS 是 Meta 的向量检索库，支持多种索引组合（IVF+PQ 等），适合离线批量检索。Qdrant 默认用 HNSW，项目没有单独调整索引类型，主要通过 `score_threshold` 和 `filter_conditions` 在查询侧做优化。

###### 4. 嵌入模型（Embedding Model）的选择对RAG效果影响很大，你在项目中是如何选择嵌入模型的？有没有做过不同模型的对比测试？

选嵌入模型主要看两点：语言匹配（中文场景必须选中文优化的模型）和任务类型（检索场景用 retrieval 变体，不要用通用模型）。搜狐实习项目用 bge-large-zh-v1.5，在中文语义检索榜上排名稳定、1024 维够用；advanced-rag 项目用 Ollama 本地嵌入，配合 SemanticChunker 做语义分块。没有做过严格对比测试，主要参考 MTEB 榜单和社区反馈选型，这是可以用 `retrieval_eval.py` 进一步量化的点。

###### 5. 什么是RAG的"上下文窗口限制"？你在项目中是如何解决这个问题的？

上下文窗口限制是指 LLM 单次接受的 token 数有上限，如果检索召回太多文段，加上对话历史和系统提示，总 token 数超限就会截断或报错。解决方式：检索后用 reranker 重排只保留相关性最高的 top-k 个块（dynamic_k 动态裁剪）；分块时控制单块大小（chunk_size=1000），保证单块不会占满窗口；系统提示精简、对话历史只保留近期；复杂问题拆成多步 Agent 分批处理，每步只传当前步需要的上下文。

###### 6. 如何提升RAG的检索精准度？除了分块、索引、重排，还有哪些方法？

查询侧：Query Rewriting（让 LLM 改写用户问题，扩展同义词/分解多跳问题）；HyDE（先让 LLM 生成假设性答案，用答案做向量检索，语义更贴近文档）；Multi-query（生成多个角度的查询并行检索再去重）。知识侧：元数据过滤（按文档 ID、时间、来源缩小搜索范围）；图谱增强（实体关系一跳邻居补充关联知识）。评估侧：用 LLM-as-a-Judge 定期评测召回质量，数据驱动调整参数，而不是靠感觉。

###### 7. 你觉得RAG目前最大的挑战是什么？如何解决Agent的幻觉问题？

最大挑战是"检索到了但没用上"——LLM 拿到正确的上下文却仍然生成错误内容，或者检索召回了不相关的噪声导致模型被干扰。幻觉解决策略：Prompt 里明确约束"如果上下文中没有答案，直接说不知道"；加来源引用让用户可以核查；用 reranker 过滤低相关性块减少噪声输入；对高置信度要求的场景用 LLM-as-a-Judge 自动校验生成结果；最终兜底是"检索失败时直接告知用户而非胡编"。

### （三）大模型与Prompt工程相关

###### 1. 你熟悉主流大模型API调用，说说你用过哪些大模型（如GPT-4/文心一言/通义千问/讯飞星火等）？它们的优缺点分别是什么？在项目中是如何选型的？

用过的：DeepSeek（推理强、国内访问快、价格低，工具调用格式偶有不规范）、通义千问（中文效果好、阿里云生态集成方便）、Groq（推理极快、免费额度大，模型能力比 GPT-4 弱）、Ollama 本地（零费用、隐私安全，但小模型工具调用能力差）、OpenRouter（聚合多家、免费模型多，稳定性参差不齐）。选型原则：复杂推理/规划任务用 DeepSeek-R1，快速工具调用用 Llama-3.1-8b，没网络条件或隐私敏感用 Ollama，想快速试用选 Groq。

###### 2. 大模型API调用的限流、鉴权机制是怎样的？你在项目中是如何处理大模型调用的超时、报错问题的？

各家 API 都用 Bearer Token 鉴权，限流一般按 RPM（每分钟请求数）和 TPM（每分钟 token 数）两个维度，超限返回 429。项目处理策略：`asyncio.wait_for(timeout=30.0)` 包裹所有 LLM 调用防止挂死；捕获到 429 时指数退避重试；捕获到兼容性错误时降级到 HTTP 直连；最终失败返回带前缀标记的错误字符串，上层做用户友好提示而不是崩溃。

###### 3. 什么是Prompt工程？你在项目中用到了哪些Prompt设计技巧（如少样本/零样本/思维链/角色设定等）？

Prompt 工程是通过精心设计输入文本来引导 LLM 产出预期输出的技术。项目里用到的：角色设定（系统提示里给 Agent 明确身份和任务边界）；思维链（ReAct 范式本身就是让模型显式输出 Thought 再 Action，强制推理过程可见）；结构化输出约束（要求模型输出固定格式的 JSON，配合括号深度匹配解析）；负向约束（"仅限授权目标"、"如无上下文请直接说不知道"）；工具描述即 Prompt（Tool 的 description 字段就是告诉 LLM 什么时候用这个工具）。

###### 4. 如何设计一个高效的Prompt来提升Agent的工具调用能力和问题解决能力？举1个你在项目中设计的Prompt案例。

关键是让 LLM 知道"有哪些工具、每个工具什么时候用、输入格式是什么"，同时强制输出格式可解析。项目里 Specialist Agent 的系统提示模板核心结构：① 角色定义（"你是一个网络渗透测试专家"）→ ② 工具列表（每个工具的名称/描述/参数说明）→ ③ 输出格式约束（"必须按 Thought:/Action:/Action Input: 格式输出"）→ ④ 边界约束（"只能操作已授权的目标"）。实测发现 `description` 里加"当用户提到 X 时使用本工具"这类触发词，能显著提升小模型的工具选择准确率。

###### 5. LLM-as-a-Judge的核心原理是什么？你在项目中是如何设计评测Prompt的？如何保证评分的一致性？

原理是用另一个 LLM 充当评委，对"问题 + 生成答案 + 标准答案"三元组打分，代替人工逐条评测。项目 `retrieval_eval.py` 里评测 Prompt 结构：给评委明确评分维度（正确性 + 完整性）、给出 0-1 分制的评分标准描述、要求输出 JSON 格式的分数和理由。保证一致性的措施：固定同一个评委模型和版本、temperature 设为 0 消除随机性、每条结果保存到文件方便人工抽检复核。

###### 6. 大模型的温度（temperature）参数有什么作用？在你的项目中，不同的任务（如规划/生成/总结）是如何设置温度参数的？

temperature 控制输出的随机性——值越低越确定性（倾向最高概率 token），越高越多样（更多随机探索）。项目里的策略：规划/工具调用用低 temperature（0~0.2），需要精确按格式输出，不能随机发挥；摘要生成用中等（0.3~0.5），要有一定的语言自然度；创意类或对话用较高（0.6~0.7），让回复不那么机械。评测（LLM-as-a-Judge）固定 temperature=0，保证打分结果可复现。

### （四）后端开发与数据库相关

###### 1. 你掌握FastAPI、Gin等Web框架，说说FastAPI的核心优势是什么？它的异步机制是怎样的？在项目中是如何利用异步提升接口性能的？

FastAPI 核心优势：基于 Python 类型注解自动生成 OpenAPI 文档、Pydantic 做请求/响应校验、原生支持 async/await 异步、性能接近 Go 框架。异步机制底层是 uvicorn + asyncio 事件循环，IO 等待时不阻塞线程，同一进程能并发处理多个请求。项目里的应用：LLM 调用用 `async def` + `await`，多个 Agent 并发执行用 `asyncio.gather`，SSE 流式推送用 `AsyncGenerator`，整个链路全异步，单进程就能支撑长时间运行的渗透任务。

###### 2. RESTful API的设计原则有哪些？你在项目中是如何设计符合RESTful规范的接口的？

核心原则：资源用名词不用动词（`/sessions` 而非 `/getSession`）；HTTP 方法语义化（GET 查、POST 创建、PUT/PATCH 更新、DELETE 删除）；状态码表达结果（200/201/400/404/500）；无状态（服务端不保存客户端会话，每次请求自带必要信息）。项目 FastAPI 路由按这个规范设计：`GET /sessions` 列会话、`POST /chat` 发消息、`GET /health` 健康检查、`POST /defense/scan` 触发扫描，聊天流式推送单独走 SSE endpoint。

###### 3. 你熟练使用MySQL、Redis，说说MySQL的索引类型有哪些？在项目中是如何创建合理的索引来提升查询效率的？

主要索引类型：B-Tree 索引（最常用，支持范围查询和排序）、Hash 索引（只支持等值查询，Memory 引擎用）、全文索引（FULLTEXT，用于文本搜索）、联合索引（多字段组合，遵循最左前缀原则）。搜狐实习项目里给舆情表的广告主、平台、发布时间建了联合索引，查询时按广告主 + 时间范围过滤，索引完全命中；只返回需要的字段不用 SELECT *，避免回表；限制时间范围防止全表扫描。最终接口响应稳定在 200ms 以内。

###### 4. Redis在你的项目中有哪些应用场景（如缓存/分布式锁/消息队列）？具体是如何实现的？

搜狐实习项目里主要用 Redis 做高频查询缓存——销售常查的热门广告主数据缓存到 Redis，TTL 设几分钟，命中缓存直接返回，未命中才走 MySQL，配合联合索引把接口响应控制在 200ms 以内，支撑 5 万次日调用峰值。分布式锁和消息队列在这个项目里没有用到，但了解原理：分布式锁用 `SET key value NX EX` 实现，消息队列可以用 List 的 `LPUSH/BRPOP` 或 Stream 数据结构实现。

###### 5. 什么是数据库的分库分表？在处理大规模数据时，你有没有考虑过分库分表？如果有，会如何设计？

分库是把不同业务的数据拆到不同数据库实例（减少单库压力），分表是把一张大表按某个维度（时间/ID 范围/Hash 取模）拆成多张小表（解决单表过大的性能问题）。搜狐项目 31 万条数据量不大，没有做分库分表，靠索引 + 缓存就够了。如果数据量到千万级，我会考虑按时间（年月）做分区表，或者按广告主 ID 取模做水平分表；跨表查询用中间件（如 ShardingSphere）屏蔽分片逻辑。

###### 6. 你用Go+Gin做过后端开发，说说Go的goroutine和channel的核心原理？在项目中是如何利用goroutine提升并发能力的？

goroutine 是 Go 运行时管理的轻量级协程，创建成本极低（初始 2KB 栈），由 GMP 调度器（goroutine/M线程/P处理器）在多核上调度，M 个 goroutine 复用 N 个 OS 线程，切换在用户态完成。channel 是 goroutine 间通信的管道，遵循"不要通过共享内存来通信，而要通过通信来共享内存"。搜狐项目里用 goroutine 并发处理多个广告主的数据聚合请求，用 `sync.WaitGroup` 等待所有子任务完成，用 channel 收集结果，整体接口吞吐量明显提升。

### （五）工程化与部署相关

###### 1. 你熟悉Git的多人协作与分支管理流程，说说你常用的Git分支模型（如Git Flow/GitLab Flow）？在项目中是如何进行代码合并、冲突解决的？

个人开源项目用简化的 Git Flow：`main` 只放稳定版本，`dev` 做日常开发，功能开 `feature/xxx` 分支，完成后 PR 到 `dev`，测试通过再合 `main` 打 tag 发版。合并优先用 `rebase` 保持线性历史，冲突时用 VS Code 的 merge editor 逐块对比解决，不确定的改动会先 `git stash` 备份再操作。搜狐实习里遵循公司规范，feature 分支 → develop → master，代码合并需要同事 review 通过才能合。

###### 2. 你熟悉Linux系统与Shell脚本，说说你在项目中用Shell脚本做了哪些工作（如部署/监控/数据处理）？举1个具体的脚本案例。

项目里写了 `build_release.sh` 做打包发布自动化：检查 Python 版本 → `uv sync` 安装依赖 → `pyinstaller` 打包成单文件可执行程序 → 打 zip 压缩包并计算 SHA256 → 输出到 `dist/` 目录。搜狐实习里写过爬虫定时调度脚本，用 crontab 每天定时触发增量爬取，并在脚本里做日志写入和失败告警（发钉钉通知）。

###### 3. Docker的核心概念（如镜像/容器/仓库）是什么？你在项目中是如何制作Docker镜像的？如何通过Docker-compose编排多容器应用？

镜像是只读的分层文件系统（类比代码），容器是镜像的运行实例（类比进程），仓库是镜像的存储和分发中心（如 Docker Hub）。项目 Dockerfile 基于 `python:3.12-slim`，复制依赖文件 → `pip install` → 复制源码 → 设置 entrypoint，多阶段构建减小镜像体积。Docker Compose 编排：`docker-compose.yml` 里定义 `app`（FastAPI 服务）、`qdrant`（向量库）、`neo4j`（图数据库）三个服务，配置端口映射、环境变量、数据卷挂载，一条 `docker-compose up -d` 启动全套环境。

###### 4. Kubernetes的核心组件（如Pod/Deployment/Service/ConfigMap）是什么？你在项目中是如何用K8s做服务部署和扩缩容的？

Pod 是 K8s 最小调度单元，包含一个或多个容器；Deployment 管理 Pod 的副本数和滚动升级；Service 提供稳定的网络访问入口（ClusterIP/NodePort/LoadBalancer）；ConfigMap 存储非敏感配置，Secret 存密钥。坦白说项目目前没有上 K8s，用 Docker Compose 单机部署。如果要上 K8s，核心改造点是：FastAPI 服务做成 Deployment + HPA 自动扩缩容，向量库 Qdrant 用 StatefulSet + PVC 持久化存储，配置通过 ConfigMap 注入，敏感信息（API Key）用 Secret。

###### 5. Nginx的反向代理和负载均衡机制是怎样的？你在项目中是如何配置Nginx的？证书配置用的是什么工具（如Let's Encrypt）？

反向代理是 Nginx 接收外部请求后转发给后端服务，客户端只看到 Nginx 不知道后端；负载均衡是把请求按策略（轮询/IP Hash/权重）分发到多个后端实例。项目里 Nginx 做反向代理把 80/443 请求转发给 FastAPI 的 8000 端口，同时处理 SSE 长连接需要配置 `proxy_buffering off` 和 `proxy_read_timeout` 调大，避免 SSE 被 Nginx 缓冲或超时断开。HTTPS 证书用 Let's Encrypt + Certbot 自动申请和续期。

###### 6. 你在项目中是如何做服务监控的？用到了哪些工具（如Prometheus/Grafana/ELK）？具体监控了哪些指标？

secbot 项目相对轻量，主要用 `/health/metrics` 端点暴露自定义指标（请求数、检索耗时、LLM 调用成功率、首 token 延迟），前端 TUI 会根据阈值展示告警提示；日志写到 `logs/agent.log`，按级别分类，工具调用的详细入参出参写到 `logs/tool_debug/`。没有接 Prometheus/Grafana，坦白说这是待完善的点。搜狐实习里用公司内部监控平台（类 Prometheus），对接口响应时间、错误率设阈值告警。

###### 7. 什么是CI/CD？你在项目中有没有做过CI/CD流水线？如果有，是基于什么工具（如Jenkins/GitHub Actions）实现的？

CI（持续集成）是代码提交后自动跑测试、构建，快速发现问题；CD（持续部署）是测试通过后自动发布到环境。项目用 GitHub Actions 实现了发布流水线（`.github/workflows/release.yml`）：打 tag 后自动触发 → 跑 `build_release.sh` 打包 Windows/Linux 两个平台的可执行文件 → 自动创建 GitHub Release 并上传产物。测试还没有完全接入 CI，这是待补充的点。

## 五、实操与场景设计类

###### 1. 请设计一个简单的电商客服AI Agent，要求支持商品咨询、订单查询、售后问题处理，说说你的整体架构和核心流程？

用 Planner-Specialist 架构：Planner 做意图识别（商品咨询/订单查询/售后），路由到三个专职 Agent。商品咨询 Agent 接 RAG 知识库（产品手册向量化）；订单查询 Agent 接订单数据库查询工具，入参用户 ID + 订单号；售后 Agent 接工单系统创建 Tool + 退款流程 Tool，高危操作（退款）需要用户确认。整体走 SSE 流式输出，保证长流程用户有实时反馈。核心难点是意图识别准确性和多轮对话上下文维护（用 ConversationBufferWindowMemory 保留近几轮）。

###### 2. 假如让你设计一个多智能体架构的代码助手Agent，支持代码生成、调试、优化、文档生成，你会设计哪些角色？角色之间如何协作？

Planner 解析用户需求 → 分发给四个专职 Agent：CodeGen（生成代码，工具：执行 sandbox）、Debugger（定位 bug，工具：静态分析 + 执行追踪）、Optimizer（性能优化，工具：profiler + 复杂度分析）、DocWriter（生成文档注释）。Coordinator 控制执行顺序：先 CodeGen 生成 → Debugger 验证 → Optimizer 优化 → DocWriter 补文档，各阶段结果传递给下阶段，Summary 汇总最终交付物。关键是 sandbox 工具隔离执行环境，防止 Agent 生成的代码污染宿主机。

###### 3. 假如你爬取的目标网站做了反爬升级，新增了验证码识别、设备指纹验证，你会如何应对？说说你的技术方案。

验证码：优先尝试 Cookie 持久化复用（首次人工完成后保存）；对于图形验证码接第三方打码平台 API（如 2captcha）；滑块类用 playwright 模拟鼠标轨迹（加贝塞尔曲线随机化）。设备指纹：用 undetected-chromedriver 绕过 WebDriver 检测；随机化 User-Agent、浏览器窗口尺寸、语言、时区等指纹字段；多账号 Cookie 池轮换分散请求特征。根本策略是"模拟真实用户行为 + 降低请求频率"，高强度反爬最终考虑 API 合作或换数据源。

###### 4. 假如你的AI Agent在实际使用中，出现了检索精准度低、生成结果不符合需求的问题，你会如何排查和优化？

分层排查：先看是检索问题还是生成问题——在 Prompt 里把检索结果直接输出给用户看，确认检索内容是否相关。检索不准：检查分块是否破坏语义完整性、调高 score_threshold 过滤噪声、试试加 reranker、检查嵌入模型是否和语言匹配。生成不准：检查 Prompt 里是否有明确约束（"只基于上下文"）、试降低 temperature、检查 context 是否被截断。最终用 `retrieval_eval.py` 量化对比，数据驱动而不是靠感觉调参。

###### 5. 假如你需要为你的多智能体RAG系统新增一个"多文档对比分析"的功能，你会如何设计？核心的技术难点是什么？

设计：用户上传/指定多个文档 → 各文档分别向量化存入独立 collection（按 doc_id 隔离）→ 新增 CompareAgent，并行对每个文档检索同一问题 → 收集各文档的检索结果 → Summary 对比异同生成结构化报告（哪个文档说了什么、主要差异在哪）。核心难点：多文档检索结果的对齐和归因（要明确标注每段内容来自哪个文档）；context 长度控制（多文档内容叠加容易超窗口，需要对每个文档单独摘要再对比）；相同概念在不同文档里表述不同时的语义对齐。

###### 6. 假如你的AI Agent服务在生产环境中出现了调用量突增、服务崩溃的情况，你会如何做容灾和扩容？

先止血：限流（Nginx rate limit 或 FastAPI 中间件）把突增流量控制住，防止雪崩；熔断（对 LLM 调用加熔断器，失败率超阈值直接返回降级响应）。排查：看监控日志确认是哪个环节崩溃（LLM 超时/数据库连接耗尽/内存 OOM）。扩容：FastAPI 服务无状态可以直接横向扩（多副本 + Nginx 负载均衡，或 K8s HPA）；LLM 调用可以接多个供应商做轮询；向量库 Qdrant 做读副本分担检索压力。长期：异步任务队列（Celery/Redis Queue）把长时间的 Agent 任务排队处理，避免请求超时。

###### 7. 请用伪代码实现一个简单的Planner-Coordinator多智能体的任务调度逻辑。

```python
async def run(user_input):
    # Planner 拆任务
    todos = await planner.plan(user_input)
    layers = get_execution_order(todos)  # 拓扑排序分层
    results = {}

    for layer in layers:
        if len(layer) == 1:
            todo = layer[0]
            results[todo.id] = await coordinator.dispatch(todo, results)
        else:
            # 无依赖的同层 Todo 并发执行
            tasks = [coordinator.dispatch(t, results) for t in layer]
            layer_results = await asyncio.gather(*tasks, return_exceptions=True)
            for todo, result in zip(layer, layer_results):
                results[todo.id] = result  # 失败也记录，不中断

    return await summary_agent.summarize(results)

async def dispatch(todo, context):
    agent = select_agent(todo.agent_hint)  # 三级路由匹配
    return await agent.execute(todo, context)
```

###### 8. 假如你需要将你的安全测试智能体适配到云原生环境，你会做哪些改造？如何保证在云环境中的安全性和可扩展性？

服务化改造：FastAPI 服务容器化，做成无状态 Deployment，会话状态从内存迁到 Redis；工具执行（nmap/扫描器）做成独立的 sidecar 容器或 Job，避免主服务权限过大。安全性：每个扫描任务跑在独立 Pod 里，任务结束自动销毁，不留痕迹；扫描工具的网络权限通过 NetworkPolicy 精细控制，只允许访问授权目标；API Key 等敏感信息用 K8s Secret 注入，不写死在镜像里。可扩展性：Agent 执行任务通过消息队列解耦，可以按负载水平扩 worker 数量。

###### 9. 假如你需要为你的AI Agent集成一个新的工具（如思维导图生成工具），说说你的集成流程和注意事项。

集成流程：继承 `BaseTool`，实现 `name`（工具唯一标识）、`description`（关键：写清楚"什么时候用/参数是什么"，决定 LLM 能否正确选工具）、`args_schema`（Pydantic 模型做参数校验）、`_arun`（异步执行逻辑）。写完后在 `ToolRegistry` 里注册，Agent 初始化时自动加载。注意事项：description 要口语化、面向 LLM 而不是面向程序员；工具执行要有超时保护；失败要返回有意义的错误描述而不是抛异常；给工具打 `sensitivity` 标签，高危操作走确认流程；写单测覆盖正常/参数缺失/执行失败三个场景。

###### 10. 假如多个子智能体在执行任务时，出现了结果不一致的情况，你会如何处理？如何保证最终结果的准确性？

结果不一致的根因通常是三类：同一资产的不同 Agent 用不同工具扫出不同结果、某个 Agent 执行失败但没有被正确标记、并发执行时上下文信息不同步。处理策略：结果按 `resource`（目标资产）维度聚合，相同资产的多个结果交给 SummaryAgent 做置信度加权（成功状态的结果权重高于失败的）；关键结论（如"存在 SQL 注入"）需要多个 Agent 或多次工具执行交叉验证才输出；最终报告里明确标注每条结论的来源 Agent 和工具，用户可以自行判断可信度；对于高风险结论，触发二次确认流程让用户决策。

## 六、软技能与工程化思维类

###### 1. 你在项目中是如何进行需求分析的？如果产品经理的需求不明确/频繁变更，你会怎么处理？

需求分析习惯先问"用户实际想解决什么问题"而不是直接实现表面需求。需求不明确时，先快速做一个最小可用版本让对方看到具体的东西，再基于反馈对齐，比反复口头讨论效率高。需求频繁变更时，关键是每次变更前问清楚影响范围，对工作量有预判，如果变更会影响已完成的核心架构，要主动提出来讨论而不是默默改，因为沉默消化会累积风险。

###### 2. 你在过往的开发中，是如何保证代码质量的？有没有做过代码评审？代码评审的核心关注点是什么？

日常用 type hints + Pydantic 做静态类型约束，关键路径写单测（尤其是工具调用和解析逻辑）。搜狐实习里有 code review 流程，核心关注点：逻辑是否正确、有没有潜在的并发/资源泄漏问题、命名是否清晰、有没有硬编码的魔法数字。开源项目自己维护，靠 GitHub Actions 跑测试 + 自己上线前走一遍关键路径验证。

###### 3. 你在项目中是如何和团队成员协作的？如果和团队成员在技术选型上产生分歧，你会怎么解决？

搜狐实习里用飞书文档对齐需求、Git 做代码协作、周会同步进度。技术选型分歧，我的处理方式是先把两个方案的优缺点和适用场景各自列出来，看哪个更符合当前项目的约束（时间/人力/运维成本），用数据或 POC 验证而不是争观点。如果还是没共识，以 leader 或有更多相关经验的人的判断为准，执行后要一起复盘效果。

###### 4. 你在开发中遇到一个技术问题，自己研究了很久都没解决，你会怎么做？

先把问题描述清楚（复现步骤、已尝试的方案、错误日志），然后优先看官方文档和 GitHub Issues，很多奇怪的 bug 别人遇到过；搜英文关键词效率比中文高很多。超过两小时没进展，主动找同事或在技术社区（Stack Overflow/Discord 频道）提问，问之前把问题上下文整理清楚，这个整理过程本身有时候就能发现问题。项目里遇到 LangChain 和国内 LLM 供应商的兼容性 bug，最终是在 LangChain GitHub Issues 里找到了根因，并自己实现了 HTTP 直连回退。

###### 5. 你如何平衡项目的开发速度和代码的工程化程度？如果项目紧急，你会牺牲哪些工程化细节？哪些是绝对不能牺牲的？

紧急时可以牺牲的：完整的文档、100% 的测试覆盖率、代码的极致优雅（能跑通优先于结构完美）。绝对不能牺牲的：核心路径的错误处理（没有兜底的异常会让系统在最糟糕的时候崩溃）、关键操作的日志和审计（出问题无法排查是更大的成本）、安全相关的校验（输入验证、权限检查不能省）。技术债要记录下来，紧急期过后及时还，不然会越堆越难改。

###### 6. 你在项目中是如何做技术文档编写的？技术文档的核心内容有哪些？

核心内容：快速开始（让新人 10 分钟内跑起来）、架构说明（各模块职责和交互关系）、API 文档（接口入参/出参/示例）、配置项说明（每个环境变量的作用和默认值）、已知问题和 FAQ。原则是文档和代码要同步更新，过时的文档比没有文档更有害。secbot 项目维护了 QUICKSTART、API.md 等文档，但坦白说项目结构部分没跟上代码迭代，这是实际存在的技术债。

###### 7. 你对"技术债务"的理解是什么？在你的项目中，有没有产生技术债务？如果有，是如何处理的？

技术债是为了短期速度主动或被动牺牲的长期质量，分"有意为之的债"（快速 MVP，知道要改）和"无意欠下的债"（认知不足导致的设计缺陷）。secbot 里有明确的技术债：授权检查没有强制串联到每次工具调用（知道但因为优先级排后了）、QUICKSTART 文档结构过时（开发推进快，文档没跟上）、小模型 function calling 适配不完整。处理方式是记录在 TODO 注释或 Issue 里，下个版本迭代时优先还，不能让债一直滚雪球。

###### 8. 你在开发中，有没有做过性能优化的工作？说说你做性能优化的思路和方法（如先定位问题，再针对性优化）。

有，搜狐实习把接口从 500ms+ 优化到 200ms 以内。思路是先定位瓶颈再优化，不要瞎猜：先加请求耗时日志，确认是数据库慢还是计算慢。定位到是 MySQL 全表扫描后，用 `EXPLAIN` 分析查询计划，补了联合索引；热点数据加 Redis 缓存，减少数据库压力；接口返回只查需要的字段，不用 SELECT *。secbot 里性能优化主要在异步化——LLM 调用和工具调用全部用 `async/await`，多 Agent 并发用 `asyncio.gather`，单进程吞吐量提升明显。

###### 9. 假如你独立负责一个AI Agent项目的开发，从0到1的整体开发流程是什么？你会如何规划时间和任务？

流程分四阶段：需求梳理（明确场景、用户、核心功能边界，1-2 天）→ 架构设计（技术选型、模块划分、接口定义，1-2 天）→ MVP 开发（核心链路先跑通：LLM 调用 + 1 个工具 + 基础 SSE 输出，1 周内）→ 功能迭代（按优先级补完工具集、多 Agent 调度、记忆系统、前端界面）。规划原则：先垂直打通一条完整链路，再水平扩展功能；每个迭代都要有可演示的结果；预留 20% 时间处理意外（LLM 兼容性问题、依赖变更等）。

###### 10. 你觉得一个优秀的AI Agent开发工程师，需要具备哪些核心素质？你自己在这些方面还有哪些提升空间？

核心素质：扎实的工程化能力（Agent 最终是工程产品，跑不起来没价值）、理解 LLM 的能力边界（知道什么场景用大模型、什么场景不适合）、快速迭代的习惯（Agent 效果高度依赖实验调参，得能快速跑通-评估-改进的循环）、系统设计能力（多 Agent 协作、数据流、容错）。我的提升空间：LLM 底层原理（Transformer 结构、训练流程）还比较浅，更多是应用层的理解；大规模分布式系统的设计经验不足，K8s 生产实战缺乏；评测体系不够系统化，目前的 LLM-as-a-Judge 方案还比较初步。
